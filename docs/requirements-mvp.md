# 小型项目管理平台 — MVP 需求说明（协作稿）

> 本文档用于对齐需求，随讨论更新。  
> **常识默认**：已按中小型内部团队的常见做法预填，可直接作为 MVP 假设。  
> **待讨论**：见 [§11](#11-待讨论与需你确认的事项)；定稿后把结论写回上文并删掉对应条。

---

## 1. 项目目标

构建一个轻量级 Web 平台，支持：

- **需求池页面**：日常需求的**堆叠收集**（人人可往里加），可标优先级，后续**关闭**或**转入开发**。
- **项目与状态流转**：需求在项目内按状态推进，成员可登录更新进度。
- **总看板**：汇总关键指标与列表，便于整体把控。

**非目标（首期不做，减少范围）**（常识默认）：

- 不做复杂甘特图、自动排期与资源均衡。
- 不做多人实时协同编辑（如飞书文档级）、不强依赖 WebSocket 推送。
- 不做可拖拽的 BPMN 审批流 / 多分支条件工作流引擎；**允许**项目经理按项目维护**线性研发状态**（名称与顺序），见 §5。
- 不做与 Jira / 禅道等的双向同步（可放第三期）。

## 2. 用户与角色

| 角色 | 说明 | 首期是否需要（常识默认） |
|------|------|--------------------------|
| **成员** | 参与项目，维护本人承接的需求、参与协作 | ✅ 是（所有登录用户默认是成员） |
| **项目经理（PM）** | 在本项目内**新建 / 调整研发状态**（名称与顺序），统筹排期与看板 | ✅ 是（可与管理员为同一人；每项目至少指定一名 PM） |
| **管理员** | 全平台用户、创建项目、把成员加入项目、归档项目 | ✅ 是（至少 1 人） |
| **项目负责人** | 业务侧统筹（可选） | ⚪ 可与 PM / 管理员兼任 |

**用户规模（约）**（常识默认）：**50 人以内**的小型团队；若更大再拆权限与性能。

**权限原则**（已对齐本轮需求）：

- 用户仅能看到**自己所属项目**内的数据（由管理员把用户加到项目成员里）。
- **研发状态**：仅 **项目经理（及管理员）** 可在该项目下**增删改状态**、**排序**（体现「评估中 → 开发中 → 测试中 → 已上线」等链路）；成员不能使用未配置的状态。
- **需求池内**：**本项目内任意登录成员**均可**新增**需求、编辑自己或他人池内条目的**标题/描述/优先级**（常识默认，避免只有 PM 能记需求）；**关闭**或**转入开发**谁可操作 — 见 §3.4。
- **研发看板内**：创建需求、改标题描述优先级等，**项目成员**一般均可；**变更研发状态**建议仅 **当前承接人** 或 **项目经理**（默认 **当前承接人 + PM 可代操作**）——见 §11 若需收紧。
- **删除需求**：建议仅管理员或「创建人 + 管理员」（实现时再二选一）。

## 3. 需求池

### 3.1 需求池页面（独立入口，已对齐本轮需求）

- 提供独立的 **「需求池」页面**（可按项目切换或从项目进入），用于**堆叠沉淀**尚未正式进研发流的需求。
- **本项目内任意登录成员**均可**新增**一条需求到池中（快速录入，鼓励随手记）；无需事先指定承接人。
- 池内列表支持按 **优先级**、创建时间等排序/筛选，便于 PM 或负责人**拉排期、清队列**。
- 单条需求在池中的后续去向只有两类主操作（见 §3.4）：**关闭**（不做/重复/否决）或 **转入正在开发**（进入研发看板）。

### 3.2 需求来源（常识默认）

- [x] **人工录入（表单）** — MVP 主路径，**主入口即需求池页面**  
- [ ] 导入表格 — 第二期  
- [ ] 其他：—

### 3.3 需求字段（常识默认）

| 字段 | 必填 | 说明 |
|------|------|------|
| 标题 | **是** | 简短可读 |
| 描述 | 否 | 支持 Markdown 可选；MVP 纯文本也可 |
| **优先级** | 否 | **P0 / P1 / P2 / P3**（或高/中/低）；在**需求池内即可标注**，进研发后仍可改 |
| 提出人 | 否 | 默认填当前登录用户，可改 |
| 期望完成日期 | 否 | 用于看板「即将逾期」提示 |
| 所属项目 | **是** | |
| **当前承接人** | 视规则 | 仅当需求**已进入研发状态**后必填语义；在池中可为空。**转入开发中**时须指定**首个承接人**（见 §3.4）。 |
| 标签 | 否 | 逗号分隔或简单多选 |
| **所处阶段** | **是** | **二选一**：仍在 **需求池** / 已进入 **研发状态**（后者取自 PM 配置列表，见 §5） |

### 3.4 池内需求的关闭与转入开发（已对齐本轮需求）

- **关闭**：将需求标记为**已关闭**（不进研发看板）；建议**必填简短原因**（不做/重复/合并到其他等）。**谁可操作**（常识默认）：**项目成员均可关闭**，或仅 **PM** — **待讨论**，见 §11。
- **转入正在开发**：将需求从池子**移入研发看板**，研发状态设为 **「开发中」**（须与 PM 在本项目配置的状态**名称一致**，或 PM 在设置里指定**「从需求池进入时的默认研发状态」**映射到「开发中」列）。**须指定首个承接人**（当前承接人），之后按 §5 在研发链上继续流转。
- 若团队流程要求先进「评估中」再进「开发中」，可拆成两步：**先入评估中**（在研发看板操作）再流转；或把「从池入库」的默认状态配成「评估中」——**待讨论**，见 §11。

**研发链上的承接人与流转**（与 §5 一致）：

- **首个承接人**：需求**从需求池第一次进入研发状态**（通常为「开发中」或 PM 指定的入库状态）时**必须指定**。
- **流转下一状态时指定其他同学**：当**当前承接人**（或 PM）将需求**流转到下一研发状态**时，须**指定下一状态的承接人**；下一承接人成为新的**当前承接人**。
- 若你希望「**只有首个承接人**在第一次往下转时能指定别人」，可在 §11 标注；**默认**：**每次**正向流转到下一状态都可指定下一承接人。

### 3.5 与「迭代 / 版本」的关系（常识默认）

- [x] **首期不区分迭代**；**需求池**与**研发中需求**同属项目下数据，用「所处阶段」区分展示。以后可加「迭代」字段或看板列。

## 4. 项目

### 4.1 项目含义（常识默认）

- [x] **一个项目包含多条需求**（需求是项目下的工作项）。

### 4.2 项目信息字段（常识默认）

| 字段 | 必填 | 说明 |
|------|------|------|
| 项目名称 | **是** | |
| 描述 | 否 | |
| 项目状态 | 否 | **进行中 / 已归档**（归档后只读或禁止新建需求） |
| 成员列表 | **是** | 管理员维护 |
| **项目经理** | **是** | 每项目至少一名（可与成员表合并为「角色：PM」标记） |

### 4.3 研发状态配置（按项目，PM 维护）

- **需求池**为**系统固定阶段**（独立页面），**不属于** PM 在研发看板里拖拽配置的那一列；从池子**转入开发**后，才进入下列研发状态。
- 每个项目有独立的 **研发状态列表**（看板列）；**项目经理**在平台内**新建、重命名、排序、删除**（删除时需处理存量需求：迁到默认状态或禁止删除有占用的状态——实现时定）。
- **示例**（仅作参考，以各项目实际配置为准）：**评估中 → 开发中 → 测试中 → 已上线**；其中 **「开发中」** 应与需求池「**转入正在开发**」按钮所落状态**一致或可映射**（见 §3.4）。
- **已上线**、**已关闭**（池内关闭）等是否算「终态」用于统计，可用配置或系统规则标记（实现时可选字段）。

## 5. 状态流转

### 5.1 状态从哪里来（已对齐本轮需求）

- **需求池**：未进入研发流的需求**只有「在池中」与「已关闭（池内）」**等系统阶段，**不占用**研发看板列。
- **研发看板**：已进入研发流的需求，其 **研发状态** 必须来自 **所属项目的研发状态配置**（§4.3）；各项目状态**名称、数量、顺序**可不同；看板按**列 = 该项目状态**展示。

### 5.2 流转规则（已对齐本轮需求）

- **从池到研**：见 §3.4（**关闭** 或 **转入开发中** + 指定首个承接人）。
- **在研发链上正向流转**：通常按 PM 配置的**顺序**进入「下一状态」（如 评估中 → 开发中 → 测试中 → 已上线）；从当前状态流转到下一状态时，**须指定下一状态的承接人**（项目内成员），见 §3.4。
- **回退 / 跳步**：是否允许非相邻状态切换 — **待讨论**（默认：**PM 可回退或纠正**；普通成员仅允许「下一状态」可减少混乱）。
- **已取消类状态**（若配置在研发链上）：进入时建议**必填简短原因**；是否必须经 PM — **待讨论**。

### 5.3 进度更新方式（常识默认）

- [x] **以研发状态为主**；不要求百分比进度条。
- [x] MVP 至少记录 **最后更新人、最后更新时间**；流转时可带**简短说明**；看板展示「最近动态」列表（轻量）。
- 建议记录 **每次状态变更：从 / 到、操作人、时间、下一承接人**（便于复盘；比全字段审计轻量）。

## 7. 总看板

### 7.1 首页优先展示（常识默认，1 最重要）

| 优先级 | 内容 | 说明 |
|--------|------|------|
| 1 | **按项目：需求池 + 各研发状态数量** | 池子堆积量与各列瓶颈一眼可见 |
| 2 | **与我相关** | **当前承接人是我**、或我提出的（可切换 Tab） |
| 3 | **风险/提醒** | 逾期、**长期未更新**（如 ≥7 天无更新且未终态）—阈值可配置第二期 |
| 4 | **最近更新** | 全平台或当前项目范围内最近 N 条 |

### 7.2 筛选维度（常识默认）

- [x] 按项目  
- [x] 按当前承接人  
- [x] 按状态  
- [x] 按日期范围（创建日 / 期望完成日）

## 8. 分期交付

| 阶段 | 范围 |
|------|------|
| **MVP** | 管理员开户、登录、项目与成员、**需求池页面**（成员可堆叠录入、标优先级）、池内**关闭 / 转入开发中**、**每项目由 PM 配置研发状态**、研发 CRUD、**流转下一状态时指定下一承接人**、研发看板、总看板统计 + 与我相关 + 最近更新 |
| **第二期** | 更细权限（如仅测试可点「测试中」）、评论与通知、导入、更完整活动流 |
| **第三期** | 报表导出、API、外部系统集成 |

**首期验收边界**（已对齐本轮需求，可再改）：

1. 管理员能创建用户、项目，并把用户加入项目；项目能指定 **PM**。  
2. 任意项目成员能在 **需求池页面** 新增需求并**标注优先级**；能对池内需求执行 **关闭** 或 **转入开发中**（转入时指定**首个承接人**，且存在与「开发中」对应的研发列或映射）。  
3. **PM 能在本项目内新建 / 排序研发状态**（至少包含 **开发中** 及后续链路，如 测试中、已上线）。  
4. **当前承接人**（或 PM）将需求在研发链上转到**下一状态**时，能**指定下一承接人**。  
5. 总看板能按项目看到 **需求池条数 + 各研发状态** 数量，并能筛到「**当前承接人是我**」的需求。  
6. 数据持久化可备份（部署时再约定）。

## 9. 技术与约束（常识默认 + 待讨论）

- **前后端偏好**：MVP 倾向 **单仓库全栈**（如 **Next.js** 或 **分离的前端 + API**），具体实现时结合你熟悉栈再定 — **待讨论**。  
- **部署环境**：**待讨论**（内网 Docker / 单机 Node / 云主机）。  
- **浏览器与移动端**（常识默认）：**PC 现代浏览器优先**（Chrome / Edge / Safari 最新两代）；**手机能查看即可**，不要求完整双列看板手势优化（第二期再加强）。

## 11. 待讨论与需你确认的事项

以下为仍可收紧或二选一的点：

1. **谁可以点「变更研发状态」**：仅 **当前承接人**，还是 **PM 也可代转**，还是 **项目内任何人都可**？（文档默认：**当前承接人 + PM**。）  
2. **流转范围**：是否**只允许**流转到「配置顺序上的下一状态」，还是允许任意状态间互转？回退谁可操作？  
3. **仅第一次交接可指定下一承接人** vs **每次进入下一状态都要指定下一承接人** —— 你原话偏向前者时，请在此写明，实现按你的口径。  
4. **需求池内谁可「关闭」**：**任意成员**还是仅 **PM / 创建人**？  
5. **从池入库**：是否**必须**落到 **「开发中」**，还是允许 PM 把默认入库状态设为 **「评估中」** 等再流转到开发中？  
6. **是否要「已阻塞」独立状态**，还是标签 + 描述即可？  
7. **登录方式**：仅用邮箱密码是否足够？是否必须 **企业微信 / 飞书 / Microsoft 365** SSO？  
8. **部署**：**纯内网** 还是公网？有无服务器 / 域名 / HTTPS？
