# TimeProject 完整版需求规格文档

> 本文档为项目管理平台的完整版需求蓝图，包含 MVP 之后的扩展功能规划。
> MVP 阶段的实际需求以 `requirements-mvp.md` 为准。

---

## 1. 引言

### 1.1 编写目的
本文档是 TimeProject 项目管理 Web 平台的需求规格说明书，用于定义系统的功能需求、非功能需求及验收标准，作为后续开发、测试和验收的基准契约。

### 1.2 产品范围
TimeProject 是一个支持项目全生命周期管理的平台，核心功能包括：
- 项目的创建与分发管理
- 需求池的累积与任务流转
- 成员抢单机制与产物上传
- 薪资记录与管理
- 全局数据看板展示

### 1.3 技术与部署架构
- **系统架构**：单租户架构，B/S 模式
- **前端适配**：采用响应式 Web 设计（H5 适配），确保移动端浏览器访问体验，为未来打包为独立 App 奠定基础
- **部署方式**：云服务器部署，采用 Docker 容器化方案
- **安全性**：HTTPS 传输、密码加密存储、JWT 鉴权

## 2. 角色与权限定义

系统包含三种核心角色，以及一种外部查看方式。

| 角色名称 | 职责描述 | 核心权限 |
|---------|---------|---------|
| **管理员** (Admin) | 系统的全局管理者，负责基础数据维护和全局监控 | - 管理所有用户和角色<br>- 管理所有项目<br>- 审批薪资发放<br>- 查看全局看板和所有人薪资 |
| **项目经理** (PM) | 项目的直接负责人，负责项目进度的推进和质量把控 | - 创建/编辑自己负责的项目<br>- 拆分需求并分发任务<br>- 验收任务产物<br>- 录入和维护团队成员薪资<br>- 查看负责项目的看板 |
| **成员** (Member) | 项目的具体执行者（如开发、测试、设计等） | - 浏览并抢单任务<br>- 更新任务状态<br>- 上传任务产物<br>- 查看个人薪资和个人任务统计 |
| **第三方** (External) | 客户或外部干系人，无需账号登录 | - 通过带 Token 的特定链接，只读查看指定项目的进度状态 |

## 3. 用户故事 (User Stories)

### 3.1 项目管理 (Project Management)

**US-PM-01：创建新项目**
> 作为**项目经理**，我希望能够**在系统中创建新项目并设置截止日期**，以便于**开始规划和跟踪新的工作目标**。
- **验收标准**：
  1. 必须填写项目名称、描述、负责人（默认为自己）、截止日期。
  2. 可选填写预算、优先级、标签。
  3. 创建成功后，项目状态默认为"规划中"。

**US-PM-02：项目状态流转**
> 作为**项目经理**，我希望能够**更新项目的状态（规划中/进行中/已暂停/已完成/已归档）**，以便于**团队和管理员了解项目的当前阶段**。
- **验收标准**：
  1. 支持状态的正向和逆向流转（如"已暂停"可恢复为"进行中"）。
  2. 状态变更时记录操作日志。

### 3.2 需求与任务管理 (Task & Flow)

**US-TK-01：需求拆分与分发**
> 作为**项目经理**，我希望能够**在项目中创建需求，并将需求拆分为具体任务放入需求池**，以便于**团队成员明确具体工作内容**。
- **验收标准**：
  1. 需求和任务采用两层结构（需求 → 任务）。
  2. 任务必须包含：标题、描述、优先级、预估工时、截止日期。
  3. 任务创建后初始状态为"待确认"。
  4. 项目经理可以将任务状态改为"待领取"，此时任务进入公共需求池可被抢单；或者直接指派给特定成员，任务进入"进行中"。

**US-TK-02：成员抢单**
> 作为**成员**，我希望能够**在需求池中查看"待领取"的任务并进行抢单**，以便于**主动承担我擅长或有空余时间处理的工作**。
- **验收标准**：
  1. 采用"先到先得"机制，点击抢单即刻生效。
  2. 抢单成功后，任务状态变更为"进行中"，执行人变更为该成员。
  3. 抢单成功触发站内信通知项目经理。

**US-TK-03：任务状态更新与产物上传**
> 作为**成员**，我希望能够**在完成任务后上传产物文件或链接，并更新任务状态**，以便于**提交我的工作成果供项目经理验收**。
- **验收标准**：
  1. 成员可以将"进行中"的任务状态变更为"待验收"。
  2. 变更状态时，必须上传附件（支持任意格式文件）或提供外部链接（如 Git 链接）。
  3. 触发站内信通知项目经理进行验收。

**US-TK-04：产物验收**
> 作为**项目经理**，我希望能够**审核成员提交的任务产物，并决定是通过还是退回**，以便于**把控项目交付质量**。
- **验收标准**：
  1. 验收通过，任务状态变更为"已完成"。
  2. 验收不通过，任务状态退回为"进行中"，必须填写退回理由（通过评论区）。

### 3.3 薪资管理 (Salary Management)

**US-SA-01：薪资录入**
> 作为**项目经理**，我希望能够**为团队成员手动录入和维护薪资记录**，以便于**根据他们的任务完成情况进行结算**。
- **验收标准**：
  1. 可以选择团队成员，输入融资金额（人民币），并可关联特定任务或项目。
  2. 录入后薪资记录状态为"待审批"。

**US-SA-02：薪资审批与发放**
> 作为**管理员**，我希望能够**查看所有"待审批"的薪资记录并进行确认**，以便于**在完成线下转账后，在系统中闭环记录**。
- **验收标准**：
  1. 管理员可以将状态修改为"已发放"。
  2. 状态变更为"已发放"时，触发站内信通知对应成员。

**US-SA-03：查看个人薪资**
> 作为**成员**，我希望能够**查看我自己的历史薪资明细和发放状态**，以便于**核对我的收入情况**。
- **验收标准**：
  1. 成员只能看到自己的薪资数据，绝对不能看到其他人的数据。

### 3.4 看板与通知 (Dashboard & Notification)

**US-DB-01：全局数据看板**
> 作为**管理员/项目经理**，我希望能够**查看项目进度、任务完成率和成员工作量等图表**，以便于**全面掌握团队和项目健康度**。
- **验收标准**：
  1. 提供核心指标卡片：项目总数、进行中项目数、延期任务数等。
  2. 提供可视化图表：任务完成率饼图、成员工作量柱状图。
  3. 支持通过下拉框筛选特定项目、特定成员或时间范围的数据。
  4. 支持将当前看板数据导出为 Excel。

**US-DB-02：第三方只读分享**
> 作为**项目经理**，我希望能够**生成一个特定项目的外部访问链接**，以便于**发给客户查看项目进度，而无需为他们创建账号**。
- **验收标准**：
  1. 生成的链接带有安全 Token。
  2. 可以设置链接的有效期限（如 7 天后失效）。
  3. 通过该链接只能看到项目的基本信息和任务流转状态，隐藏所有关于人员薪资、预算等敏感信息。

**US-NT-01：系统消息通知**
> 作为**系统用户**，我希望能够**在收到新任务、任务状态改变或被 @ 埃特时收到站内信**，以便于**及时响应工作协作**。
- **验收标准**：
  1. 页面顶部有消息铃铛图标，显示未读消息数量。
  2. 消息支持标记为已读和一键清空。
  3. 任务评论区支持输入 `@用户名` 唤起人员列表并发送提醒。

## 4. 核心业务流程图

### 4.1 任务流转核心状态机

```text
[待确认] 
   │
   ├─(项目经理指派)────────┐
   │                       │
   ▼                       ▼
[待领取] ──(成员抢单)──▶ [进行中] ◀──(验收不通过退回)──┐
                           │                           │
                           ├─(成员上传产物)             │
                           ▼                           │
                       [待验收] ───────────────────────┘
                           │
                           ├─(项目经理验收通过)
                           ▼
                       [已完成]
```

### 4.2 薪资发放流程

```text
1. 线下工作/任务完成 
   ▼
2. [项目经理] 在系统中手动录入金额 (状态: 待审批)
   ▼
3. [管理员] 审核金额是否合理
   ▼
4. [财务/管理员] 线下银行转账
   ▼
5. [管理员] 在系统中标记完成 (状态: 已发放)
   ▼
6. 系统自动发送站内信通知 [成员]
```

## 5. 数据模型设计建议 (实体关系)

主要实体及关联关系：
- **User (用户)**：关联 Role(角色)。
- **Project (项目)**：由 User(项目经理) 管理，包含多个 Requirement。
- **Requirement (需求)**：归属 Project，包含多个 Task。
- **Task (任务)**：归属 Requirement，由 User(成员) 执行，包含附件/产物链接。
- **SalaryRecord (薪资记录)**：关联 User(收款人)、User(发放人/审批人)，可关联 Task/Project。
- **Comment (评论)**：关联 Task，记录用户的讨论和退回理由。
- **Notification (通知)**：关联 User(接收人)。

## 6. 非功能性需求

1. **性能要求**
   - 页面首次加载时间不超过 3 秒。
   - 接口响应时间平均不超过 500ms。
   - 支持至少 100 人同时在线操作不卡顿。

2. **兼容性要求**
   - **PC 端**：支持 Chrome, Firefox, Safari, Edge 的最新两个大版本。
   - **移动端**：响应式适配 iOS Safari 和 Android 微信/原生浏览器。界面在竖屏模式下需保证核心操作（抢单、改状态、查看看板）的可用性。

3. **安全性要求**
   - 所有接口必须经过 JWT 鉴权（除第三方分享链接和登录接口外）。
   - 密码必须使用强哈希算法（如 bcrypt）单向加密存储。
   - 敏感操作（特别是薪资录入和发放）必须记录详细的审计日志（操作人、时间、IP、操作内容）。
   - 严格的数据越权控制（例如项目经理不能通过修改 API 参数访问其他项目的数据）。
